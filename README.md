## 커피 주문 시스템
### 프로젝트 소개

다수 서버 환경에서도 안정적으로 동작하는 커피 주문 시스템을 구현한 프로젝트입니다.
단순 기능 구현을 넘어서 동시성, 데이터 일관성, 확장성을 고려한 구조 설계에 집중했습니다.

---

### 주요 목표

- 동시 요청 환경에서도 데이터 충돌 없이 안정적으로 처리
- 주문, 포인트, 이벤트 간 데이터 일관성 보장
- 이벤트 기반 구조를 통해 확장 가능한 시스템 설계

---

### 핵심 기능
#### 1. 메뉴 목록 조회

- Redis 캐시 적용 (TTL 5분)
- DB 조회 부하 감소 및 응답 속도 개선

#### 2. 포인트 충전

- Redis 분산 락 적용 (userId 기준)
- 동시 요청 시 포인트 정합성 보장

#### 3. 커피 주문 및 결제
- 포인트 차감 + 주문 생성 + 이벤트 저장을 하나의 트랜잭션으로 처리
- Outbox 패턴을 적용하여 이벤트 신뢰성 확보

#### 4. 인기 메뉴 조회
- Redis ZSet 기반 최근 7일 집계
- 실시간 Top3 조회 가능

---

### 기술 스택

- Java 17
- Spring Boot
- Spring Data JPA
- Redis
- Kafka
- MySQL

### 설계 의사결정
#### 1. 동시성 해결

포인트는 동시에 수정될 가능성이 높은 데이터이기 때문에 Redis 분산 락을 적용했습니다.

- 락 키: lock:point:user:{userId}
- 동일 사용자 요청 → 직렬화
- 다른 사용자 요청 → 병렬 처리

-> 다중 서버 환경에서도 안전하게 동작하도록 설계

#### 2. 데이터 일관성 보장

주문 API에서는 다음 작업을 하나의 트랜잭션으로 처리했습니다.

- 포인트 차감
- 포인트 이력 저장
- 주문 생성
- Outbox 이벤트 저장

-> 부분 성공을 방지하고 실패 시 전체 롤백 보장

#### 3. Outbox 패턴 적용

DB 트랜잭션과 Kafka 전송은 하나로 묶을 수 없기 때문에 Outbox 패턴을 적용했습니다.

- 트랜잭션 내에서 이벤트를 DB에 저장
- 이후 별도 프로세스가 Kafka로 발행

-> 이벤트 유실 방지 및 재처리 가능 구조 확보

#### 4. Kafka 활용 전략

Kafka는 결제 처리에 사용하지 않고 주문 결과 전송 및 후처리 용도로 사용했습니다.

Consumer 역할:

- 주문 이벤트 로그 저장
- 인기 메뉴 집계

->  핵심 비즈니스 로직과 비동기 처리 분리

#### 5. Redis 활용 전략
캐시

- 메뉴 조회 API → Redis 캐시 적용

분산 락

- 포인트 충전 / 주문 시 동시성 제어

ZSet

- 인기 메뉴 집계 (score = 주문 횟수)

-> 조회 성능 + 동시성 제어 + 실시간 집계까지 모두 해결

#### 6. 인기 메뉴 설계

최근 7일 기준 집계를 위해  날짜별 ZSet 구조로 설계했습니다.

````
popular:menu:2026-04-01
popular:menu:2026-04-02
...
````

조회 시:

- 최근 7일 key를 union
- Top3 반환

-> 기간 기반 집계를 효율적으로 처리

### 테스트 전략
#### 1. 동시성 테스트

- 동일 userId로 다중 요청
- Redis 락으로 직렬화되는지 검증

#### 2. 주문/결제 테스트

- 포인트 차감 + 주문 생성 + 이벤트 저장 검증
- 실패 시 롤백 확인

#### 3. Kafka 통합 테스트

- Outbox → Kafka → Consumer 흐름 검증
- 이벤트 정상 처리 여부 확인

#### 4. 인기 메뉴 테스트

- Redis ZSet 점수 기반 정렬 검증
- Top3 정확성 확인

---

이 프로젝트는 단순한 CRUD 구현이 아니라
실제 서비스 환경에서 발생할 수 있는 문제를 해결하기 위한 구조 설계에 초점을 맞췄습니다.

- 동시성 → Redis 분산 락으로 해결
- 데이터 일관성 → 트랜잭션 + Outbox로 보장
- 확장성 → Kafka 기반 이벤트 구조로 확보

-> 안정성과 확장성을 동시에 고려한 시스템을 설계하는 경험을 얻을 수 있었습니다.
